Error-state principles (locked across all 5)

What every error does

  • Soft red (gap/launch) or amber (warning) — never aggressive crimson
  • One-line problem statement in plain language, no jargon
  • One sentence of cause / why (when we can be specific)
  • Primary action: retry or continue what you can
  • Secondary action: escape to safer ground (back to hub)
  • Request reference code (small, monospace) for support tickets
  • Reassurance: "this doesn't affect your record" / "nothing is lost" — candidate's biggest anxiety in a job-prep app

What none of them do

  • No stack traces, no API URLs, no HTTP codes shown to the candidate
  • No modal dialogs that block the page — banners or inline cards only
  • No "contact support" without a reference code attached
  • No "your data may be lost" language unless it's actually true
  • No empty white error pages — always offer two actions

For v1 we ship these five. Mid-call audio errors (mic permission denied, browser unsupported) live inside the meeting client — out of scope for our React app. Auth failures from PCS deep-link bounce back to PCS — also out of scope here.
